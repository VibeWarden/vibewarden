Reflected XSS  High Mitigated by VibeWarden

User-supplied input is reflected directly into the HTML response without escaping, allowing an attacker to inject and execute arbitrary JavaScript in the victim's browser.

What is Reflected XSS?

Reflected Cross-Site Scripting (XSS) occurs when a web application takes input from an HTTP request and immediately includes it in the response without sanitisation or escaping. The malicious script is "reflected" off the server back to the user's browser, where it executes in the context of the vulnerable site.

An attacker crafts a URL with a malicious payload in a query parameter and tricks a victim into clicking it — via phishing email, social media, or a short link. Because the script comes from a trusted domain, the browser executes it with full access to cookies, local storage, and the DOM.

Common consequences include: session hijacking, credential theft, defacement, and redirecting users to malicious sites.

The Vulnerable Code

The endpoint /vuln/xss-reflected?q=... in this demo app is intentionally written without output encoding:

// INTENTIONALLY VULNERABLE — demonstrates reflected XSS
fmt.Fprintf(w, "<p>Search results for: %s</p>", r.URL.Query().Get("q"))

When a browser requests /vuln/xss-reflected?q=hello, the response contains <p>Search results for: hello</p>. If the value of q is <script>alert('XSS')</script>, that script tag lands verbatim in the HTML and the browser executes it — unless a Content Security Policy prevents it.

VibeWarden Protection

Content Security Policy blocks inline script execution

VibeWarden adds the following header to every response from your app:

Content-Security-Policy: default-src 'self'; ...

Because default-src 'self' does not include 'unsafe-inline', the browser refuses to execute any inline <script> tags injected by the attacker. The malicious HTML is written into the page, but the script never runs.

You can verify this in the browser's developer console: when you submit the XSS payload, you will see a CSP violation message such as:

Refused to execute inline script because it violates the following Content Security Policy directive: "default-src 'self'"

Important: CSP is a defence-in-depth measure. The correct fix at the application layer is to always HTML-encode output — e.g. use html/template in Go instead of fmt.Fprintf. CSP protects against mistakes; it is not a substitute for correct encoding.

How to Fix This in Your App

Use Go's html/template package, which escapes output by default:

// SAFE: html/template escapes all values automatically
tmpl := template.Must(template.New("").Parse(`<p>Search results for: {{.}}</p>`))
tmpl.Execute(w, r.URL.Query().Get("q"))

Never use fmt.Fprintf to write user-supplied data into an HTML response. If you must build HTML strings, call html.EscapeString() on every untrusted value first.
