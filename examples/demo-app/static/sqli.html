SQL Injection  High Not Mitigated by VibeWarden

Unsanitised query parameters are interpolated directly into SQL statements, allowing an attacker to read, modify, or delete arbitrary data in the database.

What is SQL Injection?

SQL injection (SQLi) occurs when an application builds SQL queries by concatenating user-supplied input directly into the query string without sanitisation or parameterisation. An attacker who controls the input can terminate the intended query and append their own SQL clauses, changing the query's semantics entirely.

Classic consequences include: bypassing authentication, dumping entire tables, modifying or deleting records, and (in some databases) executing operating-system commands. SQLi has consistently ranked in the OWASP Top 10 for over a decade.

The Vulnerable Code

The endpoint GET /vuln/sqli?user=... in this demo app builds its query like this — intentionally wrong:

// INTENTIONALLY VULNERABLE — string concatenation, no parameterisation
query := "SELECT id, user_id, title, content FROM notes WHERE user_id = '" + userParam + "'"
rows, _ := db.Query(query)

For a legitimate request ?user=admin the generated SQL is:

SELECT id, user_id, title, content FROM notes WHERE user_id = 'admin'

But when the attacker sends ?user=admin' OR '1'='1, the SQL becomes:

SELECT id, user_id, title, content FROM notes WHERE user_id = 'admin' OR '1'='1'

The condition '1'='1' is always true, so the database returns every row in the table — including rows belonging to other users.

VibeWarden Protection

Honest answer: VibeWarden does NOT prevent SQL injection

SQL injection is a vulnerability in application code, not in the network layer. VibeWarden sits between the internet and your app as a reverse proxy — it has no visibility into the SQL queries your app runs internally.

VibeWarden provides two partial mitigations that reduce the blast radius but do not prevent the attack:

  • Authentication (Ory Kratos): Unauthenticated attackers cannot reach protected routes at all. This eliminates anonymous exploitation, but any authenticated user — including one with a stolen session token — can still inject SQL.
  • Rate limiting: Automated scanners and exfiltration scripts that make many requests per second will be throttled. This slows down, but does not prevent, a patient attacker.

The real fix is in your application code. Use parameterised queries. Full stop.

How to Fix This in Your App

Replace string concatenation with a parameterised query. The database driver handles escaping — your code never touches the value:

// SAFE: parameterised query — user input never touches the SQL string
rows, err := db.QueryContext(ctx,
    "SELECT id, user_id, title, content FROM notes WHERE user_id = ?",
    userParam,
)

With parameterised queries the database engine receives the query structure and the parameter value separately. Even if userParam contains SQL syntax, it is treated as a plain string literal — never as part of the query structure.

Additional layers of defence:

  • Apply the principle of least privilege: the database user your app connects with should only have the permissions it actually needs (SELECT on the notes table, not DROP TABLE).
  • Use an ORM or query builder that parameterises automatically (e.g., sqlc, sqlx, GORM).
  • Validate and allowlist input where possible — reject strings that contain characters your app has no legitimate reason to accept.
