⚠️ The endpoint POST /vuln/csrf-transfer is INTENTIONALLY VULNERABLE. It accepts transfers with no CSRF token check. Do not deploy this in production.

CSRF  Medium Mitigated by VibeWarden

Cross-Site Request Forgery tricks an authenticated user's browser into sending a forged request to your app. The app acts on it because the session cookie is automatically included by the browser.

What is CSRF?

When a user is logged in, their browser automatically attaches the session cookie to every request to that domain — including requests triggered by visiting a third-party site. An attacker can host a page that silently sends a POST request to your app, and if the app has no CSRF protection, the request succeeds.

Attack steps

  1. Victim logs in to the target app (session cookie is set).
  2. Victim visits an attacker-controlled page in the same browser session.
  3. The attacker page silently submits a form to /vuln/csrf-transfer.
  4. The browser includes the session cookie; the server processes the transfer.

No user interaction is required — the form can auto-submit on page load.

Simulated Attacker Page

What an attacker page would contain

An attacker would host a page like this. The form targets your app's transfer endpoint and auto-submits when the page loads. If the victim is logged in and the app has no CSRF protection, the transfer goes through.

<!-- attacker-csrf.html — hosted on evil.example.com -->
<body onload="document.forms[0].submit()">
  <form action="https://your-app.example.com/vuln/csrf-transfer"
        method="POST" style="display:none">
    <input name="to" value="attacker">
    <input name="amount" value="1000">
  </form>
</body>

VibeWarden Protection

SameSite cookies and Referrer-Policy

VibeWarden configures Ory Kratos to set session cookies with SameSite=Strict. This means the browser will NOT include the session cookie when the request originates from a different site.

Set-Cookie: ory_session=...; SameSite=Strict; HttpOnly; Secure

With SameSite=Strict, a cross-origin POST to /vuln/csrf-transfer will arrive without the session cookie. VibeWarden's auth middleware then rejects the unauthenticated request before it even reaches the app handler.

VibeWarden also sets Referrer-Policy: strict-origin-when-cross-origin, which limits the information exposed in the Referer header on cross-origin requests — useful for server-side origin validation.

Defence in depth: even if SameSite cookies are bypassed (e.g. on older browsers), adding a CSRF token at the application layer provides a second line of defence.
